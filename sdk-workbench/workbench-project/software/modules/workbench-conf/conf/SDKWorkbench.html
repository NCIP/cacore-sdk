Overview

The caCORE SDK Workbench is a graphical user interface (GUI) created as a companion to the caCORE SDK Toolkit.  It is intended to centralize and facilitate the management of the SDK Code Generation and Deployment processes for one or more SDK projects. 

In order to do so, the Workbench needs access to a compatible SDK toolkit instance, version 4.2 or newer.  The Workbench will take the user through a step-by-step process to download and install a compatible version of the SDK software on a local machine.  This relieves the user from having to locate and visit multiple websites, and from having to read the related SDK Toolkit download, installation, and configuration instructions.  By using the Workbench, the end-user is freed to focus on modeling their domain, versus learning the intricacies and nuances of operating the SDK Toolkit.

Additional Workbench functions and features include:

·         The elimination of the need to manually edit the SDK Toolkit configuration properties or the need to manually invoke the Toolkit processes that use these properties.

·         The grouping of SDK Toolkit code generation and deployment properties into intuitive, logical property units, displayed as distinct tabs within a window panel.  This facilitates the enabling and disabling of entire property groupings, such as the Writable API or Security property tabs, based upon whether corresponding properties (e.g., Enable Writable API? or Enable Security?) are selected or not.

·         The implementation of real-time property validation prior to project generation and deployment.  This includes validation between related properties, or even groups of properties.  The user effectively no longer has to worry about property validation rules, as the Workbench will dynamically determine and indicate which properties are required under given conditions.  Property format is also enforced, where applicable.

·         The implementation of general SDK usage and help functions, including extended property descriptions.

 

 

Launching the Workbench

The Workbench is available as a Java Web Start application, or as a standalone application.

Launching the Web Start Version       

Java Web Start is a framework developed by Sun Microsystems, which allows users to start application software for the Java Platform directly from the Internet using a web browser.  To launch the Web Start version of the Workbench, open a web browser, and enter the following link in the address bar to the Workbench Java Network Launching Protocol (JNLP)[1] file:

 @WEBSTART_LINK@ e.g.:  http://localhost:8080/workbench/Workbench.jnlp

Opening and executing the Workbench.jnlp file will cause a security warning similar to the following to be displayed:

Click the Allow button.  The Workbench requires that its jars be signed so that it has adequate access rights to client-side project folders and resources.  The Workbench packaging process uses a dynamically generated key and self-signed certificate to sign the workbench jars, which is the source of the warning.

The first time the Workbench Web Start application is downloaded, a dialog window will be displayed which contains a progress bar indicating the download process: 

This dialog box should not be displayed during subsequent startup of the Web Start version, as the Workbench application is cached by the Java Web Start framework following the initial download.

Once the Workbench has been successfully downloaded, another security warning will be displayed, as shown:

 Once again, the Workbench packaging process uses a dynamically generated key and self-signed certificate to sign the workbench jars, which is the source of the warning.  Click the Run button.  An SDK Workbench logo will be displayed, followed by the initial Workbench screen.  This completes the Web Start startup process.

Launching the Standalone Version

To launch the standalone version of the Workbench, execute the "run" target within the primary Workbench Ant build script, /workbench-project/software/build/build.xml of the Workbench distribution.  Execution of this build script target will compile and package the required Workbench artifacts, and invoke the Workbench initialization process.

Installing an SDK Instance

The Workbench facilitates the management of the SDK Code Generation and Deployment processes for one or more SDK projects.  To do so, it is dependent upon a compatible SDK toolkit instance, which the Workbench can also install.  The process is detailed below.

Selecting the SDK Installation Directory

To install a Workbench-compatible version of the SDK Toolkit, start by either clicking the Install SDK button located at the top-left portion of the Workbench toolbar, or select the Install SDK option from the File drop-down menu.  A single tab window will be displayed, requesting that the destination SDK installation directory be selected or specified, as shown below:

Either enter the absolute path to the directory in the Select SDK Installation Directory field, or click the Browse button to the right of the field.  Clicking the Browse button will cause a Select Directory dialog window to appear, as shown below:

Either navigate to and select an existing directory, or create a new directory by clicking the Create New Folder icon on the top-right hand portion of the dialog toolbar.  Click the Open button to complete the directory selection process.

Installing the SDK

Once the SDK installation directory has been selected, the Install button will become enabled at the bottom of the window.  Click it to begin the SDK Toolkit installation process.  A progress bar with the words Installing the SDK will be displayed during the installation process.  The bar will disappear once the installation process has completed.

Creating a caCORE-like Application

The Workbench provides the ability to both set SDK code generation properties and generate a caCORE-like application based upon these properties. 

Selecting the Project Generation Directory

To get started with the code generation process, either click the Create Application button located on the Workbench toolbar, or select the Create Application option from the File drop-down menu.  As shown below, a dialog window will pop up, requesting that the Project Generation directory be selected or specified:

At this point, either navigate to and select an existing directory, or create a new directory by clicking the Create New Folder icon on the top-right hand portion of the dialog toolbar.  Click Open to complete the project generation directory selection process.

Managing Code Generation Properties

Once the Project Generation directory has been selected, click the Open button.  The resulting Create Application window is shown below:

 

The Create Application window contains six (6) tabs.  The first five tabs, Project, Model, Code Generation, Writable API, and Security, facilitate the management of the SDK application generation properties.  The properties displayed on these tabs correspond to logical sections within the SDK Toolkit v4.2 codegen.properties file.  The sixth tab, Generate Application, summarizes the properties entered in the first five tabs, and also controls the launch of the application generation process.  Each tab is further discussed in turn below.

Project Tab Properties

The first tab, Project, is shown above.  It contains high-level, project-wide properties, which include:

·         Project Directory:  the directory where the project will be generated;

·         SDK Install Home Directory:  the directory containing an SDK Toolkit instance, v4.2 or newer, which is required by the Workbench;

·         Project Name:  Used to create (name):

o   Output project directory folder name;

o   Beans JAR file name;

o   ORM JAR file name;

o   Client JAR file name;

o   WAR file name;

o   Web Service Namespace;

o   Documentation title in the generated API (Javadocs);

·         Namespace Prefix[2]:  used to create:

o   XSD's (Schema's) ;

o   XML Mapping and Un-mapping files;

o   Locate the UML Project in caDSR when adding Hibernate Validator annotations in generated domain objects.  Use format: gme://<<Classification Scheme (UML Project ShortName)>>.<<Context>>/<<Classification Scheme Version>>/<<<Classification Scheme Item> (Optional)>>;

·         Web Service Name[3]: used to create the:

o   Web service name in the Web Services deployment descriptor (WSDD).

o   Web service URL, which uses the Web service name as a suffix  (i.e., ${SERVER_URL}/services/${WEBSERVICE_NAME})

Model Tab Properties

The second tab, Model, is shown below.  It contains properties related to the selection and processing of the project model file:

Properties on this tab include:

·         Model File:  specifies the absolute path of the file which contains the object/data model.  Either enter the path or select it using the Browse button.

·         Model File Type:  specifies whether the object/data model file was created using Enterprise Architect (EA) or ArgoUML (ARGO).  This property is automatically set by the Workbench based upon the file type extension of the model file property.

·         Logical Model: specifies the name of the base package within the object/data model that contains the logical model data.

·         Data Model: specifies the name of the base package within the object/data model that contains the data model data.

·         Include Package[4]: specifies the regular expression (java.util.regex) pattern(s) of any fully qualified package names within the object/data model that should be processed by the code generator. Separate patterns with a comma (',') as a delimiter. NOTE:  This property applies only to original package names as defined within the UML Logical Model, and NOT to any GME Namespace package name(s) that may be present in the model as Tag Values.

·         Exclude Package: specifies the regular expression (java.util.regex) pattern(s) of the fully qualified package name(s) within the object/data model that should be ignored (not processed) by the code generator.  Use a comma (',') as a delimiter to separate patterns.  Note:  All packages are first filtered (constrained) by the INCLUDE_PACKAGE property value, and then further filtered by the EXCLUDE_* values.  NOTE:  This property applies only to original package names as defined within the UML Logical Model, and NOT to any GME Namespace package name(s) that may be present in the model as Tag Values.

·         Exclude Name: specifies the regular expression (java.util.regex) pattern(s) of the fully qualified class name(s) within the object/data model that should be ignored (not processed) by the code generator.  Use a comma (',') as a delimiter to separate patterns.  NOTE:  This property applies only to original fully qualified class names as defined within the UML Logical Model class names, and NOT to any GME Namespace class name(s) that may be present in the model as Tag Values.

·         Exclude Namespace: specifies the regular expression (java.util.regex) namespace pattern of the fully qualified package name(s) within the object/data model that should be ignored (not processed) by the code generator.  Use a comma (',') as a delimiter to separate patterns.  NOTE:  Unlike the previous three properties, this property applies ONLY to any GME Namespace values that may be present in the model as Tag Values.  In this case, the original NAMESPACE_PREFIX property defined within this file is effectively "overridden".

Code Generation Tab Properties

The third tab, Code Generation, is shown below.  It contains properties that control the code generation process; i.e., properties that are used to enable or disable code generation step(s). These properties accept values of either 'true' or 'false'. Setting the value to 'false' for a component disables the code generation of that component, while setting the value to 'true' enables it:

 

Properties on this tab include:

·         Validate Logical Model:  if enabled, checks the validity of the object model. 

·         Validate Model Mapping:  if enabled, checks the validity of the object-to-database mapping

·         Generate Hibernate Mapping: if enabled, generates Hibernate related artifacts

·         Generate Beans:  if enabled, generates Java beans from the object model

·         Generate Castor Mapping:  if enabled, generates castor mapping files for the object model

·         Generate XSD:  if enabled, generates the XSD and XML Mapping files

·         Generate WSSD:  if enabled, generates the Web Services Deployment Descriptor (WSDD) file

·         Generate Hibernate Validator:  if enabled, will generate Hibernate Validator annotations using caDSR Permissible Values to create an enumeration of valid values.  Annotations are added during code generation to domain object attributes.  At runtime, the value of domain object attributes containing annotations will be validated prior to insert or update.  This setting is only valid if the Writable API extension (Enable Writable API Extension?) and security (Enable Security?) properties are enabled as well.

·         Generate XSD with GME Tags:  If enabled, uses GME Namespace tag values when generating Castor Mapping and XSD files.  For more information, see GME Namespace Wiki: https://wiki.nci.nih.gov/display/caCORE/GME+Namespace

·         Generate XSD with Permissible Values:  If enabled, uses caDSR Permissible Values as restriction/enumeration elements for a given attribute of a class when generating XSD files.  This setting is only valid if the Generate XSD property is enabled as well.

Writable API Tab Properties

The fourth tab, Writable API, is shown below.  It contains properties that enable and control the Writable API Extension:

Properties on this tab include:

·         Enable Writable API Extension: if enabled, will change the SDK generated application service interface and corresponding implementation to the Writable API.

·         Database Type: A suffix appended to certain tag value keys (e.g., NCI_GENERATOR.<database-type>, NCI_GENERATOR_PROPERTY.<database-type>) added to the primary key columns within the UML model.  The tag values are used by the Hibernate Mapping file transformer to generate primary key settings for a given class.  The database type suffix is necessary when supporting multiple databases through the same UML model.  Only relevant if the Writable API extension is enabled.

·         CLM Project Name: used to populate the (Common Logging Module) CLM logging table (LOG_MESSAGE) application column. Only valid if CLM and the Writable API are both enabled.

·         Hibernate Identity Generator Tag:  if using system-wide primary key generator settings, the value for the primary key generator class.  Only relevant if the Writable API extension is enabled.

·         Enable Common Logging Module:  if checked, enables the CLM.  Only valid if the Writable API extension is enabled.

Security Tab Properties

The Fifth tab, Security, is shown below.  It contains properties that enable and control security related options:

Properties on this tab include:

·         Enable Security: Used to enable or disable security within the generated system.  This applies to all of the SDK interfaces, including:

o   Web Interface (GUI)

o   Java API Interface (local and remote clients)

o   Writable API

o   Web Service Interface

·         Enable Instance Level Security:  Used to enable/disable CSM instance level security.  Only relevant if Security is enabled.

·         Enable Attribute Level Security:  Used to enable/disable CSM attribute level security.  Only relevant if Security is enabled.

·         CSM Project Name:  Used as a prefix when creating the CSM security configuration file name.  Only relevant if Security is enabled.  The CSM Project Name value must match the project application name used when configuring security using CSM's UPT.

·         Cache Protection Elements:  Indicates whether or not CSM Protection Elements should be cached.  Only relevant if Security is enabled.

·         Enable caGRID Login Module:  Used to enable or disable the caGRID Login Module.  Only relevant if Security is enabled.

·         SDK Grid Login Service Name:  Enter the SDK GRID Login Service name.  Only relevant if Security and the caGRID Login Module are both enabled.

·         caGRID Login Module name:  Enter the caGRID Login Module name.  Only relevant if Security and the caGRID Login Module are both enabled.

Application Creation Panel Buttons

The Previous and Next buttons facilitates movement from one active[5], or enabled, tab to another.  The user can also navigate between tabs by directly clicking on the desired tab. 

The Save button stores the entered or selected properties on all of the tabs back to the SDK codegen.properties file.  The Save button only becomes enabled whenever a code generation property is changed.  The Workbench allows properties to be saved even if there are validation errors.  However, all validation errors will need to be resolved, and the properties re-saved, prior to generating the project.

Generating an Application

This section discusses the sixth and final Create Application panel tab, Generate Application. 

Generate Application Tab

As mentioned in the previous section, The Create Application window contains six (6) tabs.  The last of these tabs, Generate Application summarizes the properties entered or selected in the first five tabs, and also controls the launching of the code generation process based upon these properties.  An example is shown below:

On this screen, only relevant, selected properties are shown.  As an example, the Writable API tab contains some six properties.  However, since the Enable Writable API Extension? property has been set to false, none of the other properties are shown on this screen.  The same applies to the Security section. 

The Generate button initiates the SDK Code Generation process.  It only becomes enabled whenever there are no pending property changes that need to be saved, there are no validation errors, and the Generate Application tab is selected.  Once clicked, the application will be generated using the properties summarized on the Generate Application tab.

If the selected Project Generation directory is not empty, a dialog warning that existing contents will be overwritten is displayed, and the user must confirm whether or not to proceed, as shown below:

If the Yes button is clicked, the code generation process will continue, and any previously generated content will be overwritten.  If the user selects the No option, the code generation process is terminated and the user is returned to the Generate Application tab.

Finally, if any errors are encountered during the code generation process, a dialog is displayed indicating that this is the case, and the user is requested to check the logging output for details.

 

Deploying a Generated Application

The Workbench provides the ability to manage both the setting of SDK deployment properties and the deployment of a previously generated SDK application based upon these properties. 

Selecting the Project Generation Directory

To get started with the deployment process, either click the Deploy Application button located on the Workbench toolbar, or select the Deploy Application option from the File drop-down menu.  As shown below, a dialog window will pop up, requesting that the Project Generation directory be selected:

At this point, either navigate to and select an existing directory, or create a new directory by clicking the Create New Folder icon on the dialog toolbar. 

Managing Deployment Properties

Once the Project Generation directory has been selected, click the Open button.  The resulting Deploy Application window is shown below:

 

 

The Deploy Application window contains nine (9) tabs.  In the case of a local deployment, the properties displayed on these tabs correspond to logical sections within the SDK Toolkit v4.2 install.properties file; or in the case of a remote deployment , the {remote-environment}-install.properties file.

Deploy Type Tab Properties

The first tab, Deploy Type, is shown above.  Initially, it is the only tab that is enabled.  This tab is used to specify whether the deployment type is Local or Remote, and if remote, what the remote environment should be.  This tab is also used to load to the workbench the corresponding local or remote set of deployment properties. 

Properties on this tab include:

·         Deployment Type:  Indicates whether the deployment is Local or Remote.  If the local deployment option is chosen, the application will be deployed to a local application server (Tomcat or JBoss) instance, depending on which properties are subsequently specified on the App Server tab.  If the Remote deployment option is chosen, the application will be deployed to the specified remote environment, as detailed below.

·         Remote Environment:  Indicates which of the five (5) remote environments should be targeted.  Only relevant if the deployment type is set to Remote.  Options include:

o   Development;

o   Quality Assurance (QA);

o    Stage

o   Production;  or

o    Other

Loading a Set of Deployment Properties

Once the deployment type has been specified, the Load Properties button will become active.   Deployment property sets do not get automatically loaded.  The Load Properties button must be explicitly clicked before the corresponding properties are loaded to the Workbench.

App Server Tab Properties

The second tab, App Server, is shown below.  It contains properties which specify and configure the target application server.  If unsure, these properties can be left with their default values:

Properties on this tab include:

·         Server Type:  Indicates whether deploying to a JBoss or Tomcat Application Server instance, which are the only two options available at this time.

·         Tomcat/JBoss Hostname:  The hostname (or sitename) that uniquely identifies the Tomcat or JBoss instance on the network ;

·         Tomcat/JBoss AJP Port:  Tomcat or JBoss Apache JServ Protocol (AJP)[6] Port Number;

·         Tomcat/JBoss HTTP Port:  Port Tomcat or JBoss will listen to for HTTP[7] requests. 

·         Tomcat Shutdown Port: Port Tomcat will listen to for application server shutdown requests

·         JBoss JNDI Port:  Port JBoss will listen to for JNDI[8] requests.

·         JBoss SSL Port:  Port JBoss will listen to for SSL[9] requests.

DB Tab Properties

The third tab, DB, is shown below.  It contains properties which specify and configure the primary data model database server:

Properties on this tab include:

·         Database Type:  Indicates whether deploying to a MySql or Oracle database instance, which are the only two options available at this time.

·         Use a JNDI-based Connection:  Whether or not to use a JNDI-based database connection.  If selected, the database properties will need to be specified via a standalone server JNDI configuration file.

·         DB JNDI Name:  The JNDI name used in the server JNDI configuration file to lookup and identify the database configuration properties.  Only relevant if the Use JNDI-based Connection is selected.

·         DB Connection URL:  The database connection URL.  This is a compound property which is composed of the DB Hostname, Port, and Schema properties.  It is not modifiable, and is dynamically updated by the Workbench as the dependent properties are updated.  This property is provided as a convenience to the user

·         DB Hostname:  The hostname (or sitename) that uniquely identifies the DB instance on the network.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Port:  The port number the database instance is listening to for data requests.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Schema: The database schema name.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Username:  The username used to authenticate database requests.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Password:  The password used to authenticate database requests.  Only relevant if the Use a JNDI-based Connection is not selected.

Logging Tab Properties

The fourth tab, Logging, is shown below.  It contains properties which configure the database instance for the Common Logging Module (CLM).  The Logging tab is only enabled if the Enable Common Logging? option was selected and saved on the Writable API tab during the code generation process:

Properties on this tab include:

·         Database Type:  Currently MySql is the only database type supported by the Common Logging Module.

·         DB Connection URL:  The CLM database connection URL.  This is a compound property which is composed of the CLM DB Hostname, Port, and Schema properties.  It is not modifiable, and is dynamically updated by the Workbench as the dependent properties are updated.  This property is provided as a convenience to the user.

·         DB Hostname:  The hostname (or sitename) that uniquely identifies the CLM DB instance on the network

·         DB Port:  The port number the CLM database instance is listening to for data requests;

·         DB Schema: The CLM database schema name

·         DB Username:  The username used to authenticate CLM database requests;

·         DB Password:  The password used to authenticate CLM database requests.

CSM DB Tab Properties

The fifth tab, CSM DB, is shown below.  It contains properties which configure the CSM database server for the Common Security Module (CSM).  The CSM DB tab is only enabled if the Enable Security Extension? option was selected on the Security tab during the code generation process:

Properties on this tab include:

·         Use DB Connection Settings:  Whether or not to use the same property settings as specified on the DB tab.  If selected, all property values will be copied from the DB tab properties.  Any sub-sequent changes to one or more of the DB tab properties will be automatically reflected to properties on this tab as well.  Also, when selected, the remaining DB properties are disabled for user input, as the Workbench will ensure that the properties remain synchronized.

·         Database Type:  Indicates whether deploying to a MySql or Oracle CSM database instance.  MySql and Oracle are the only two options available at this time.

·         Use a JNDI-based Connection:  Whether or not to use a JNDI-based database connection.  If selected, the database properties will need to be specified via a standalone server JNDI configuration file.

·         DB JNDI Name:  The JNDI name used in the server JNDI configuration file to lookup and identify the database configuration properties.  Only relevant if the Use JNDI-based Connection is selected.

·         DB Connection URL:  The database connection URL.  This is a compound property which is composed of the DB Hostname, Port, and Schema properties.  It is not modifiable, and is dynamically updated by the Workbench as the dependent properties are updated.  This property is provided as a convenience to the user

·         DB Hostname:  The hostname (or sitename) that uniquely identifies the DB instance on the network.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Port:  The port number the database instance is listening to for data requests.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Schema: The database schema name.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Username:  The username used to authenticate database requests.  Only relevant if the Use a JNDI-based Connection is not selected.

·         DB Password:  The password used to authenticate database requests.  Only relevant if the Use a JNDI-based Connection is not selected.

caGRID Auth Tab Properties

The sixth tab, caGRID Auth, is shown below.  It contains properties which configure the caGRID Authentication Login Module.  The caGRID Auth tab is only enabled if the Enable Security Extension? option and the Enable caGRID Login Module? options were selected on the Security tab during the code generation process:

Properties on this tab include:

·         caGRID Environment:  Specifies the caGRID v1.2 environment that the generated application should connect to.  Options include:

o   Development 1.2

o   Training 1.2

o   Quality Assurance (QA)

o   Stage 1.2

o   Production 1.2

·         caGRID Security Certificate File:  Used to authenticate the application when logging into the caGRID.  Either enter the absolute file path to the caGRID Security Certificate file, or select it by using the Browse button to the right of the property.

·         caGRID Security Key File:   Used to authenticate the application when logging into caGRID.  Either enter the absolute file path to the caGRID Security Key file, or select it by using the Browse button to the right of the property.

 

Advanced Tab Properties

The seventh tab, Advanced, is shown below.  It contains miscellaneous advanced properties, and typically would not need to be modified by the end-user:

Properties on this tab include:

·         Application Base Linux Path:  The base Linux path to where the application server and generated application artifacts will be deployed to.  Though required, only relevant if the deployment Operating System (OS) is UNIX-based.

·          Application Base Windows Path:  The base Windows path to where the application server and generated application artifacts will be deployed to.  Though required, only relevant if the deployment Operating System (OS) is Windows-based.

·         EHCache Path:  The location of the EHCache[10].

Remote SSH Tab Properties

The eighth tab, Remote SSH, is shown below.  The eighth tab, Remote SSH, contains the properties required when connecting and deploy to a remote environment via Secure Shell[11] (SSH).  This tab is only enabled if the Remote deployment option is specified:

Properties on this tab include:

·         SSH Server Username:  The authentication username when connecting via the SSH protocol.

·         SSH Hostname:  The hostname (or sitename) that uniquely identifies the remote server instance on the network.

·         SSH Port:  The SSH Port configured on the remote server.

·         SSH Key File:  Enter the SSH protocol Key file path, or select it using the Browse button to the right of the property.

·         SSH Temp Directory:  A temporary directory on the remote server that can be used while deploying the application server and generated application artifacts.

 

Application Deployment Panel Buttons

The Previous and Next tabs facilitates movement from one active[12], or enabled, tab to another.  The user can also navigate between tabs by directly clicking on the desired tab. 

The Save button stores the entered or selected properties on all of the deployment tabs back to the SDK install.properties file (or {remote-environment}-install.properties file, if the remote deployment option is chosen).  The Save button only becomes enabled whenever a deployment property is changed.

Deploying a Previously Generated Application

This section discusses the ninth and final Deploy Application panel tab, Deploy Application. 

Deploy Application Tab

Configuring Workbench Logging

The Workbench uses Apaches Log4j, a Java based logging utility[13], to log both Workbench and SDK Toolkit related messages.  By default, the logger is configured to send messages to the console (standard output).  Also, the Log Level is set to INFO. 

To change logging settings for the standalone version of the Workbench, modify the /workbench-project/software/modules/workbench-conf/conf/log4j.properties file, and then re-launch the standalone version of the Workbench.  Logging levels cannot be changed for the Web Start version, as the Workbench log4j properties file is packaged within one of the self-signed Workbench Jars distributed via the Web Start framework.  If you would like to redirect logging to a file instead of standard out, change the FileAppender setting (currently set to c:/log/client.log) to the desired value.  Also, if you would just like to log more detailed information, change the logging level from INFO to DEBUG.
